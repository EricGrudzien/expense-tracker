# Chat Feature — Specification & Implementation Plan

## 1. Purpose

Add a conversational chat page to the Expense Tracker. Users type natural-language questions
about their expense data (e.g. "How much did I spend on airline in April?") and receive
answers generated by Amazon Bedrock (Claude), backed by live SQL queries against `expenses.db`.

---

## 2. Architecture

```
Browser                    Flask Backend                  Amazon Bedrock
───────                    ─────────────                  ──────────────
User types question  ──►  POST /api/chat
                           │
                           ├─ Build system prompt
                           │   (DB schema + category list + rules)
                           │
                           ├─ Call Bedrock converse API ──────────►  Generate SQL
                           │                              ◄──────── SQL in response
                           │
                           ├─ Validate SQL (must be SELECT)
                           ├─ Execute SQL against expenses.db
                           │
                           ├─ Call Bedrock converse API ──────────►  Format answer
                           │   (user question + SQL + raw results)   from results
                           │                              ◄──────── Natural-language answer
                           │
                           └─ Return JSON response ──►  Display in chat UI
```

### Request/Response flow

1. Frontend sends `POST /api/chat` with `{ "message": "..." }`
2. Backend builds a system prompt with the DB schema, valid categories, and safety rules
3. Backend calls Bedrock (first call) asking the model to generate a SQL SELECT query
4. Backend validates the SQL (must start with SELECT, no DDL/DML)
5. Backend executes the SQL against `expenses.db` (read-only)
6. Backend calls Bedrock (second call) with the original question, the SQL, and the raw
   query results, asking the model to produce a clear human-readable answer
7. Backend returns `{ "answer": "...", "sql": "...", "data": [...] }` to the frontend
8. Frontend displays the answer as a chat bubble with an expandable "Show SQL" section

## 3. Technology Choices

| Decision              | Choice                                | Rationale                                    |
|-----------------------|---------------------------------------|----------------------------------------------|
| LLM                   | Claude Sonnet 4 (`us.anthropic.claude-sonnet-4-20250514`) | Good SQL generation quality, fast |
| Bedrock API           | `converse` API via `boto3`            | Simpler than `invoke_model`, model-agnostic  |
| AWS region            | `us-east-1` (configurable)            | Common Bedrock availability region           |
| Conversation mode     | Single-turn (stateless)               | Simpler v1; each request is independent      |
| SQL extraction        | Text-based (parse from code block)    | Simpler than tool-use for v1                 |
| Credentials           | Default boto3 credential chain        | Env vars, ~/.aws/credentials, instance role  |

## 4. Backend Specification

### 4.1 New dependency

Add `boto3` to `backend/requirements.txt`.

### 4.2 New endpoint: `POST /api/chat`

**Request body:**
```json
{ "message": "How much did I spend on hotels last month?" }
```

**Response 200:**
```json
{
  "answer": "You spent $1,200 on hotels across 3 expense reports.",
  "sql": "SELECT SUM(se.amount) AS total FROM sub_expenses se JOIN expense_reports er ON er.id = se.report_id WHERE se.category = 'hotel'",
  "data": [{ "total": 1200.0 }]
}
```

**Response 400:** `{ "error": "Message is required" }`
**Response 500:** `{ "error": "..." }` for Bedrock or SQL execution failures.

### 4.3 System prompt (first Bedrock call — SQL generation)

The system prompt must include:
- The full schema of all three tables (`categories`, `expense_reports`, `sub_expenses`)
- Column names, types, and relationships (foreign keys)
- The current list of valid category slugs and their display labels (queried live from DB)
- A note that `expense_reports.date` is stored as `YYYY-MM-DD` text
- A note that report totals are derived (SUM of sub_expenses.amount), not stored
- Rules:
  - Generate ONLY a single SQL SELECT statement
  - No INSERT, UPDATE, DELETE, DROP, ALTER, CREATE, or any DDL/DML
  - Use SQLite syntax
  - Return the SQL inside a markdown code block (```sql ... ```)
  - If the question cannot be answered from the database, say so instead of generating SQL

### 4.4 Second Bedrock call — answer formatting

After executing the SQL, send a second prompt containing:
- The original user question
- The SQL that was executed
- The raw query results (as JSON)
- Instruction: "Format these results as a clear, concise answer to the user's question.
  Use dollar formatting for monetary amounts. Do not include the SQL in your answer."

### 4.5 Safety guardrails

| Guard                     | Implementation                                              |
|---------------------------|-------------------------------------------------------------|
| SQL validation            | Strip whitespace, check that it starts with `SELECT` (case-insensitive). Reject everything else. |
| Read-only execution       | Open a separate connection, execute inside a try/except, always rollback (never commit). |
| Query timeout             | Set a timeout on the SQLite connection (e.g. 5 seconds).    |
| No schema mutation        | The system prompt explicitly forbids DDL/DML. The code also rejects non-SELECT. |
| Error handling            | If SQL generation fails or query errors, return a friendly error message, not a stack trace. |
| Input length              | Reject messages longer than 1000 characters.                |

### 4.6 SQL parsing

Extract SQL from the model response by:
1. Look for a ```sql ... ``` code block
2. If not found, look for a ``` ... ``` code block
3. If not found, try to use the entire response text (trimmed) as SQL
4. Validate the extracted SQL starts with SELECT

## 5. Frontend Specification

### 5.1 New files

| File            | Purpose                          |
|-----------------|----------------------------------|
| `chat.html`     | Chat page HTML                   |
| `chat.js`       | Chat page JavaScript             |
| `chat.css`      | Chat-page-specific styles        |

### 5.2 Navigation

Add a "Chat" tab to the nav bar on all 4 pages (index.html, query.html, categories.html,
chat.html). Chat tab is active on `chat.html`.

### 5.3 Chat UI layout

```
┌──────────────────────────────────────┐
│  💸 Expense Tracker    [Reports] ... │  ← header + nav
├──────────────────────────────────────┤
│                                      │
│  ┌──────────────────────────────┐    │
│  │  Chat messages area          │    │  ← scrollable, grows to fill
│  │  (scrolls vertically)        │    │
│  │                              │    │
│  │  [assistant bubble]          │    │
│  │           [user bubble]      │    │
│  │  [assistant bubble]          │    │
│  │    ▸ Show SQL                │    │  ← expandable
│  │                              │    │
│  └──────────────────────────────┘    │
│                                      │
│  ┌──────────────────────────────┐    │
│  │ Ask about your expenses...   │ ➤  │  ← input bar
│  └──────────────────────────────┘    │
└──────────────────────────────────────┘
```

### 5.4 Message types

| Type       | Alignment | Style                                    |
|------------|-----------|------------------------------------------|
| User       | Right     | Indigo background, white text            |
| Assistant  | Left      | Light gray background, dark text         |
| Error      | Left      | Red-tinted background                    |
| Loading    | Left      | Animated dots ("Thinking…")              |

### 5.5 "Show SQL" toggle

Each assistant message that includes SQL shows a clickable "▸ Show SQL" link below the
answer text. Clicking it expands to reveal the SQL in a monospace code block. Clicking
again collapses it. The toggle text changes to "▾ Hide SQL" when expanded.

### 5.6 Behavior

- On page load: show a welcome message from the assistant ("Ask me anything about your
  expenses — e.g. 'What's my total spending?' or 'Show all airline costs'")
- User types a message and presses Enter or clicks Send
- A user bubble appears immediately
- A loading indicator ("Thinking…") appears while waiting
- On response: the loading indicator is replaced with the assistant's answer
- The chat area auto-scrolls to the bottom after each new message
- The input is cleared and re-focused after sending
- If the request fails, show an error bubble
- Conversation history is kept in the DOM only (no persistence, lost on page refresh)

### 5.7 Input validation

- Empty messages are ignored (button disabled when input is empty)
- Messages longer than 1000 characters show an inline error
- Input and Send button are disabled while a request is in flight

## 8. Open Questions / Future Enhancements

These are NOT in scope for v1 but worth noting:

- **Multi-turn conversation**: Send conversation history for follow-up questions
  ("What about just in March?" after asking about airline costs)
- **Streaming responses**: Use Bedrock streaming to show the answer as it generates
- **Query result tables**: Render tabular data inline in the chat when the result has
  multiple rows/columns
- **Suggested questions**: Show clickable example questions on the empty state
- **Rate limiting**: Throttle chat requests to avoid excessive Bedrock costs
- **Caching**: Cache identical questions to avoid redundant Bedrock calls
